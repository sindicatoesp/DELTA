- id: P0-6
  title: "Fix hardcoded /en/ URL in email notifications"
  file: app/backend.server/models/user/invite.ts
  line: 53
  root_cause: >
    Email notification functions (invite, welcome, password reset) construct
    URLs via ctx.fullUrl(), which calls urlLang(this.lang, path) where this.lang
    is the language of the HTTP request that triggered the email — typically the
    admin's current UI language. When an English-speaking admin invites a
    French-speaking user, the invite link contains /en/ regardless of the
    recipient's locale. The clientLogger.ts file also contains a separate
    hardcoded /en/ at line 165 (this.endpoint = `${baseUrl}/en/api/client-log`)
    for the client-side logging endpoint. The fix for emails is to derive the
    URL language prefix from the recipient user's stored locale preference (or a
    system default env var) rather than from the calling handler's language context.
  fix_complexity: Simple
  requires_spec: false
  requires_test: true
  fix_summary: >
    Pass the recipient user's stored language preference to invite/email
    functions so ctx.fullUrl() uses the recipient's locale, not the admin's;
    fall back to a DEFAULT_LANGUAGE constant if no preference is stored.
    Separately fix the hardcoded /en/ in app/utils/clientLogger.ts line 165.
  antipattern: null
